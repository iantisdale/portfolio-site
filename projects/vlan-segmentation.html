Case Study: Split DNS Architecture and Dependency Reduction

This project came from a practical operations problem: infrastructure systems should not be fully dependent on an internal DNS path that may itself be unavailable during an outage, reboot, or recovery event.

Problem

Some infrastructure devices and core services were positioned to depend too heavily on internal DNS. That creates a bootstrap problem: when DNS is degraded, the systems needed to recover the environment may also lose resolution.

Approach

Separated DNS responsibilities by device role. Infrastructure nodes were planned to use either public DNS only or AdGuard Home with a public secondary resolver, while client devices would receive AdGuard Home only through Dynamic Host Configuration Protocol (DHCP).

Technical Decisions
  • Keep infrastructure recovery paths independent.
  • Avoid circular dependencies between DNS and core management services.
  • Use AdGuard Home as the primary filtered resolver for clients.
  • Reserve public fallback for infrastructure only.
Outcome

The environment moved toward a more resilient split-DNS model that better supports recovery, troubleshooting, and clearer role separation between infrastructure and client systems.

1. Identify the operational risk

Core systems should not fail to recover because they depend on the same name resolution plane that is currently broken.

2. Separate infrastructure from clients

Infrastructure devices get independent resolution options. Clients get consistent filtered DNS through the internal resolver layer.

3. Reduce ambiguity

Clarifying which devices should and should not have public fallback helps prevent accidental policy drift and future troubleshooting confusion.
